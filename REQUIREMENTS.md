## 0. 本書の位置づけ

本書は、日本の建設業向け**工事原価管理 Web アプリケーション**を新規開発するための要件定義書である。既存の市販パッケージソフトウェア（MARS Evo / NEXT 等）の **業務カバー範囲は参考にする**が、画面・データ構造・コードはすべて**独自設計（クリーンルーム方式）**とする。

設計はすべて**業界標準の業務概念**（実行予算・内訳書・代価表・出来高査定・労務出面・機械出面・外注管理など、いずれも業界共通用語）に基づく。

## 1. プロジェクト概要

### 1.1 背景

- 現在、社内で建設業向けパッケージ（MARS）を使用して工事原価管理を行っている。
- パッケージは**スタンドアロン Windows アプリ**であり、同時利用ライセンス数に制約がある。
- 結果として、**現場社員・経理担当・経営層など全員が業務データに常時アクセスできない**状況である。
- データ自体は社内に蓄積されており、活用余地が大きい。

### 1.2 ゴール

| # | ゴール |
|---|---|
| G1 | **全社員が Web ブラウザから工事原価データに常時アクセスできる**こと |
| G2 | 工事原価管理に必要な業務（予算・出来高・労務・外注・支払・入金・帳票）を**Web で完結**できること |
| G3 | 現場からスマートフォン／タブレットで**日報・出面の入力**ができること |
| G4 | 既存業務データを移行し、**過去工事の参照・分析**ができること |
| G5 | アトラス（積算ソフト）等の外部データ取り込みに対応すること |

### 1.3 非ゴール（あえて作らない）

- 既存パッケージとの**画面互換**（操作感は独自に最適化する）
- 既存パッケージとの**ファイル形式互換**（独自フォーマット）
- 会計ソフト・給与ソフトの**機能内包**（連携 API は将来検討）
- マルチテナント SaaS 化（**単一企業内利用**を前提とする）

### 1.4 ターゲットユーザ

| 役割 | 想定人数 | 主な利用シーン |
|---|---|---|
| 経営層 | 数名 | 全工事の収支ダッシュボード閲覧 |
| 工事部長・現場代理人 | 10〜30名 | 実行予算編成、出来高査定、外注管理 |
| 現場監督・職長 | 30〜100名 | 作業日報入力、出面入力、写真・進捗報告 |
| 経理担当 | 数名 | 支払・入金管理、帳票出力 |
| 積算担当 | 数名 | アトラス取り込み、内訳書・代価表編成 |
| 閲覧者 | 全員 | 自分の関わる工事の状況確認 |

## 2. スコープ（段階的開発）

### 2.1 Phase 1（MVP）：閲覧・集計・帳票の Web 化【約3〜4ヶ月】

- 工事一覧・工事詳細の表示
- 実行予算・内訳書・代価表の**閲覧**
- 出来高・労務・機械・外注実績の**閲覧**
- 主要帳票の Excel / PDF 出力
- ユーザ認証・権限管理
- データ取り込み（既存業務データの初期投入）

### 2.2 Phase 2：末端業務の Web 入力【+ 約6ヶ月】

- 作業日報の入力（PC・モバイル両対応）
- 労務出面・機械出面の入力
- 出来高査定の入力
- 写真・コメント添付
- 通知（メンション・承認依頼）

### 2.3 Phase 3：コア業務の Web 入力【+ 約12ヶ月】

- 実行予算編成（内訳書・代価表・次代価表の編集）
- 単価マスタ・歩掛マスタ管理
- アトラス DBF 取り込み
- 外注発注書・注文請書発行
- 支払依頼・入金消込

### 2.4 スコープ外（本書範囲外）

- 工程管理（ガントチャート）
- CAD 連携
- 経理仕訳・財務会計
- 給与計算
- 受注前の見積機能（積算は外部ソフト＝アトラス側で行う前提）

### 3.4 内訳書・代価表データ構造（業界標準）

```
工事 (project)
 └─ 実行予算 (budget_header) ※世代管理
     └─ 内訳明細 (budget_item) ※階層構造
         ├─ 工種・科目・要素分類
         ├─ 名称・規格・単位・数量・単価・金額
         └─ 代価表 (unit_price_table) ※項目に紐づく単価内訳
             └─ 代価明細 (unit_price_item) ※さらに階層化可
                 └─ 次代価表 (sub_unit_price_table) ※再帰的
```

## 4. 非機能要件

### 4.1 パフォーマンス

| 項目 | 要件 |
|---|---|
| 同時利用ユーザ数 | 100ユーザ（初期）／ 500ユーザ（将来） |
| ページ応答時間 | 主要画面 95%tile で 2秒以内 |
| 帳票出力時間 | 標準帳票 10秒以内、大型帳票 30秒以内 |
| 一覧表示 | 1万件以上の一覧は仮想スクロール対応 |

### 4.2 セキュリティ

| 項目 | 要件 |
|---|---|
| 通信暗号化 | HTTPS（TLS 1.2 以上）必須 |
| 認証 | パスワードハッシュは Argon2id |
| 認可 | 全 API でロール・工事権限チェック |
| 監査ログ | データ更新・削除・帳票出力をすべてログ記録 |
| パスワードポリシー | 12文字以上、英数記号混在 |
| セッション | HttpOnly / Secure / SameSite=Strict Cookie |
| CSRF | Double Submit Cookie |
| XSS | フレームワーク機構＋ CSP ヘッダ |
| SQL Injection | Prisma（パラメタライズドクエリ） |
| ファイルアップロード | ウイルススキャン・拡張子＆ MIME 検証 |

### 4.3 可用性

| 項目 | 要件 |
|---|---|
| 稼働時間 | 平日 7:00〜22:00 で 99.5%以上 |
| バックアップ | 日次フル＋トランザクションログ4時間毎 |
| データ保持期間 | 過去7年分（建設業法準拠） |
| 障害時 RTO | 4時間以内 |
| 障害時 RPO | 4時間以内 |

### 4.4 操作性

- **画面密度**：業務 PC ユーザ向けに高密度（Excel ライクなテーブル UI）
- **モバイル対応**：日報・出面・写真添付など現場機能はレスポンシブ
- **キーボード操作**：内訳編集はキーボードだけで完結
- **多言語**：日本語のみ（v1）
- **アクセシビリティ**：WCAG 2.1 AA を可能な範囲で

### 4.5 拡張性

- 機能追加が容易なモジュラー設計（NestJS のモジュール分割等）
- 将来の API 公開を想定した REST API 設計
- マスタ項目の動的拡張（カスタムフィールド）

### 4.6 監査ログ

| 対象 | 記録内容 |
|---|---|
| ログイン・ログアウト | ユーザ・時刻・IP・User-Agent |

| 帳票出力 | 帳票種類・対象工事・出力者・出力日時 |
| 権限変更 | 変更前後の権限・実施者 |

## 9. API 設計

### 9.1 設計方針

- **REST** ベース、リソース指向
- すべて `/api/v1/...` プレフィックス
- 認証：Cookie ベース（NextAuth セッション）
- レスポンス：JSON（成功時は `data`、エラー時は `error`）
- ページング：`?page=1&limit=50` または `cursor`
- フィルタ：`?filter[status]=in_progress&filter[year]=2026`
- ソート：`?sort=-created_at`（`-` で降順）
- 部分応答：`?fields=id,name,status`
- バリデーションエラー：HTTP 422 + `{ error: { fields: [...] } }`
- 業務エラー：HTTP 409 / 422 + `{ error: { code, message } }`

## 13. リスクと対策

| # | リスク | 影響 | 対策 |
|---|---|---|---|
| R1 | データ移行の不整合 | 高 | 段階的移行、移行前後の数値突合テスト |
| R2 | 内訳ツリー編集 UI の難度 | 高 | Phase 3 まで編集機能を先送り、まず閲覧で価値提供 |
| R3 | 帳票の体裁要求が厳しい | 中 | 既存帳票の社内サンプルから新規 Excel テンプレ作成、現場とすり合わせ |
| R4 | 既存パッケージとの並行運用での二重入力 | 中 | Phase 1 は読み取り専用なので発生せず、Phase 2 以降はマスタ同期方針を別途定義 |
| R5 | モバイル現場利用の通信環境 | 中 | オフライン対応（PWA + 同期）を Phase 2 で検討 |
| R6 | 法改正（インボイス・電帳法等）対応 | 中 | 帳票テンプレートを差し替え可能に設計 |
| R7 | ユーザの ICT リテラシ差 | 中 | 操作マニュアル整備、現場説明会、ヘルプアイコン |
| R8 | ベンダロックインの再発 | 低 | OSS 中心の技術選定、データは標準形式で常時エクスポート可能に |

## 16. 付録：Claude Code での実装指針

### 16.1 進め方の推奨

1. **本書の確認**：本書を Claude Code セッション開始時に読み込ませる（`CLAUDE.md` から参照）
2. **段階的実装**：Phase 1 のマイルストーンに沿って週次でタスクを切る
3. **データモデル先行**：Prisma スキーマと型定義（packages/schemas）を最初に固める
4. **API → フロント**：API を先に作りモックで動かしてからフロントを実装
5. **テスト同時並行**：機能実装と同時に Vitest を書く
6. **PR 単位を小さく**：1機能 1 PR、Claude Code でレビュー可能なサイズに

### 16.2 各モジュール実装順序（Phase 1）

1. `auth` モジュール（ログイン・セッション）
2. `users` モジュール（CRUD）
3. `customers` モジュール
4. `projects` モジュール
5. `budgets` モジュール（読み取りのみ）
6. `progress` モジュール（読み取りのみ）
7. `daily-reports` モジュール（読み取りのみ）
8. `attendance` モジュール（読み取りのみ）
9. `reports` モジュール（帳票生成）
10. `imports` モジュール（初期データ投入）
11. `audit` モジュール

### 16.3 Claude Code との対話のコツ

- 大きな機能は「ドメインモデル設計 → Prisma スキーマ → サービス層 → コントローラ → フロント」と分けて依頼
- スキーマ変更は必ず Prisma マイグレーションファイルを作る
- UI 変更は shadcn/ui のコンポーネントから選ぶ
- 業務ロジックは `services/` 配下に集約、コントローラは薄く保つ
- 帳票のような複雑な機能は ADR（アーキテクチャ決定記録）を `docs/adr/` に残す
